Contributions are welcome. This page covers everything you need to get started: environment setup, code style, common tasks, and debugging.

## Development setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/kalil0321/reverse-api-engineer.git
    cd reverse-api-engineer
    ```
  </Step>

  <Step title="Install dependencies">
    The project uses [uv](https://github.com/astral-sh/uv) for dependency management:

    ```bash
    uv sync
    ```
  </Step>

  <Step title="Install Playwright browsers">
    ```bash
    playwright install chromium
    ```
  </Step>

  <Step title="Run locally">
    ```bash
    uv run reverse-api-engineer
    ```
  </Step>
</Steps>

## Linting and static analysis

All checks are available individually or via the lint script:

<CodeGroup>

```bash all checks
./scripts/lint.sh
```

```bash ruff linter
uv run ruff check src
```

```bash ruff formatter
uv run ruff format src
```

```bash mypy
uv run mypy src
```

</CodeGroup>

The project follows [PEP 8](https://peps.python.org/pep-0008/) and uses ruff for both linting and formatting.

## Building

Use the clean build script to remove stale bytecode caches before building. This prevents version display bugs caused by cached `.pyc` files:

```bash
./scripts/clean_build.sh
```

For a manual build without cleaning:

```bash
uv build
```

```bash

```

## Code style conventions

### Type hints

Use type hints throughout. Import from `typing` for compatibility with Python 3.11:

```python
from typing import Optional, Dict, Any, List

def get_run(run_id: str) -> Optional[Dict[str, Any]]:
    ...
```

### Async/await

Engineers use `async`/`await` for LLM streaming. Keep async boundaries consistent — do not mix sync and async code in the same call chain without explicit bridging.

### UI output

Use [Rich](https://github.com/Textualize/rich) for all terminal output. Do not use plain `print()` statements:

```python
from rich.console import Console

console = Console()
console.print("[green]Run complete![/green]")
```

### Error handling

Wrap all external calls (LLM API, browser, filesystem) in `try`/`except` with user-friendly error messages. Avoid bare `except` clauses:

```python
try:
    response = await client.generate(prompt)
except anthropic.APIConnectionError as e:
    console.print(f"[red]Connection failed: {e}[/red]")
    raise
```

### Configuration migration

When renaming config keys, add backward compatibility logic in `config.py`. The project supports automatic migration of old config files to new key names.

## Common tasks

### Adding a new model

1. Add pricing to `src/reverse_api/pricing.py` in the `MODEL_PRICING` dict (if the model is not already covered by LiteLLM)
2. Update model choices in `src/reverse_api/tui.py` → `get_model_choices()` if the model should appear in the settings menu
3. Document the model in `README.md` under the model selection section

### Adding a new agent provider

1. Add provider logic to `src/reverse_api/browser.py` → `run_agent_browser()`
2. Add the new config fields to `src/reverse_api/config.py` → `DEFAULT_CONFIG`
3. Update the settings menu in `src/reverse_api/cli.py`
4. Document the provider in `README.md` under agent configuration

## File structure

```
src/reverse_api/
├── cli.py               # Main entry point, mode cycling, slash commands
├── base_engineer.py     # Abstract base for SDK implementations
├── engineer.py          # Claude SDK implementation
├── opencode_engineer.py # OpenCode SDK implementation
├── browser.py           # Playwright HAR capture + agent mode
├── config.py            # Configuration manager
├── session.py           # Run history and metadata
├── messages.py          # Message persistence
├── pricing.py           # Model pricing database
├── tui.py               # Terminal UI (Rich-based)
├── opencode_ui.py       # OpenCode-specific UI extensions
└── utils.py             # Path helpers, run ID generation
```

Runtime data lives under `~/.reverse-api/`:

```
~/.reverse-api/
├── config.json                  # User configuration
├── history.json                 # Run history with costs
└── runs/
    ├── har/{run_id}/            # Captured traffic per run
    ├── scripts/{run_id}/        # Generated API clients per run
    └── messages/{run_id}.jsonl  # LLM conversation logs
```

## Debugging

| Technique | How to use |
| --- | --- |
| OpenCode debug logs | Set `OPENCODE_DEBUG=1` before running |
| Stagehand logs | Suppressed by default. Controlled via `_suppress_stagehand_logs()` in `browser.py` |
| CLI stderr noise | Minified JS stack traces are filtered to a single line. Set `DEBUG=1` for full output |
| Message logs | All LLM interactions are saved to `~/.reverse-api/runs/messages/{run_id}.jsonl` |

```bash
OPENCODE_DEBUG=1 uv run reverse-api-engineer
```

## Pull request process

1. Create a feature branch from `main`
2. Make your changes and run `./scripts/lint.sh`
3. Test locally with `uv run pytest`
4. Submit a pull request with a clear description of what changed and why

For bug reports, open a GitHub issue and include:
- Steps to reproduce
- Operating system and Python version
- Full error message and traceback

## Release process

The single source of truth for versioning is `pyproject.toml`. `src/reverse_api/__init__.py` reads the version dynamically via `importlib.metadata` — do not edit it manually.

See `RELEASING.md` in the repository for the full release checklist.
